Harden the flight-sim actuator boundary: fail-safe on non-finite, physical saturation limits

Two safety defects in the flight-sim actuator/controller stack:

- NaN latching (K17-A2). FirstOrderActuator.step clipped the command straight into
  self.state, so a single NaN/inf command permanently poisoned the actuator state and
  every delayed measurement after it. IsoFluxController.pid_step accumulated the raw
  error into err_sum, so one NaN error latched the integrator forever.
- Non-physical saturation (K17-A4). The actuator saturation defaulted to +/-1e9 A and
  the controller's actuator_current_delta_limit to 1e9 A -- an ineffective gate that
  bounds nothing a real coil would ever see.

Fixes:
- FirstOrderActuator.step now treats a non-finite command as a fault: it holds the
  last valid state (fail-safe hold), advances the delay line so measurement timing
  stays consistent, counts the event in a new faults attribute, and never latches the
  bad sample into self.state.
- pid_step returns a safe zero command on a non-finite error and leaves err_sum /
  last_err untouched, so a NaN can never poison the integrator.
- Default saturation limits move to +/-5e4 A (50 kA, the ITER PF/CS coil-current scale,
  consistent with the 1 MA/s rate limit over a ~50 ms control cycle), so an anomalous
  command is bounded to a physical current. Sim commands are O(1), so the flight-sim
  trajectory is unchanged; the bound now actually catches runaway commands.

tokamak_flight_sim.py stays at 100% line coverage; seven new tests cover the fail-safe
hold (no poison after NaN/inf), the integrator guard, and the physical default limits.
The gym environment (which builds IsoFluxController with default limits) is unaffected.

Closes KIMI due-diligence findings K17-A2 (actuator + PID-integrator NaN-latching arcs)
and K17-A4 (unphysical actuator limits, Python side).
