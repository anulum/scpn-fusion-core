Vectorise the phase-kernel cos/sin fill with a runtime-dispatched AVX2 path

After the algebraic rewrite each oscillator costs exactly two
transcendentals — its cos θ and sin θ — so the phase kernels are now
bounded by that fill. `f64::sin`/`f64::cos` are two independent scalar
libm calls per element, each with its own argument reduction, and the
compiler cannot vectorise across them.

Add a `sincos` module: one Cody-Waite reduction to [−π/4, π/4] feeds two
cephes minimax polynomials, computed in straight-line FMA arithmetic
(ternaries lower to selects, sign flips to ±1 multiplies), so the whole
`(sin, cos)` pair comes from a single branch-free expression. `fill_pairs`
runs it under an `is_x86_feature_detected!("avx2","fma")` gate in a
`#[target_feature]` clone (four f64 lanes), with a scalar fallback — the
baseline build stays plain x86-64, so the wheel never issues an
unsupported instruction on an older CPU (no SIGILL). Accuracy is within a
few ulp of the libm reference over the phase range (tested to <1e-15),
comfortably inside the 1e-12 cross-tier gate and the 1e-14 order-parameter
gate; the 10 phase parity cases pass unchanged.

Also make the per-step rayon split coarse: the fill and the element update
now run over fixed 2048-wide `par_chunks` instead of a fine-grained
`par_iter`. The per-step tasks are small, so fine-grained splitting
over-subscribes and slows the batched run on many-core hosts (measured:
kuramoto_run fell from ~4.8x at 4 threads to ~2.3x at 12); coarse chunks
bound the fork-join count regardless of core count and let each chunk's
arithmetic update auto-vectorise.

Authoritative CI baseline after the preceding algebraic commit
(4-vCPU runner, numpy < 2.0): kuramoto_run 4.76x, upde_run 6.44x,
upde_tick 6.14x, kuramoto_step 2.88x. This commit's figure is measured by
the phase-kernels-benchmark CI artifact; local (numpy 2.2.6, 4 threads)
regression tracks kuramoto_run ~4.8x and upde_run ~7.6x on that harsher
floor, so upde_run is expected to approach the M-3 ≥10x bar on CI.
