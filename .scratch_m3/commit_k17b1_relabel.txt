Relabel the SCPN/PID/MPC lane honestly as a P-gain readout sanity check

The benchmark was titled "SCPN vs PID/MPC Benchmark" and framed as a controller
comparison, but the SCPN lane is configured with delay_ticks=1024 (>> run length),
so the timed Petri-net transitions never fire: the control action is the readout of
the injected features scaled by the readout gains -- a static proportional (P-gain)
map, not the network's spiking dynamics. The PID/MPC gains are fixed baselines, not
tuned optima, and the pass thresholds explicitly license the SCPN lane to be 1.10x /
1.35x worse. Presenting this as SNN-vs-PID/MPC evidence is not honest.

Make the lane self-describe what it is:
- Retitle to "SCPN readout-path sanity lane vs PID/MPC baselines" (module docstring,
  rendered markdown title).
- Add lane_semantics + claim_boundary fields to the campaign output and a claim-boundary
  line to the markdown: the SCPN action is a static P-gain readout (transitions inert),
  the PID/MPC are fixed baselines, and passes_thresholds is a plumbing/sanity band, NOT
  evidence the SNN is competitive. Point to the disturbance-rejection lane where the SNN
  loses to MPC by orders of magnitude.
- Strengthen the inline comment at the transition definitions to say the same.
- Document `seed` honestly as a compatibility no-op (the lane is structurally
  deterministic; seed never seeds an RNG, so there is no run-to-run variance) (K17-B2).

pass (10) and now assert the honest lane semantics + claim boundary. The committed
